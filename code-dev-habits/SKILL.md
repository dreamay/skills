# 代码开发习惯

## 何时使用

本 skill 用于**系统化熟悉和整理一个代码项目**。典型触发语包括：

- "熟悉下这个项目"
- "先了解这个项目"
- "分析下这个代码库"
- "给我做个项目 onboarding"
- "这个项目是做什么的"
- "整理下项目文档"

如果用户的话里包含"熟悉"、"了解"、"分析"、"onboarding"、"整理文档"、"todo"等关键词，优先使用本 skill。

## 目标

1. 快速建立对项目的全局认识（技术栈、目录、核心类、入口）
2. 识别当前代码中的待完善点、潜在风险、可改进项
3. 生成或更新 `TODO.md`
4. 按需更新 `README.md` / `CLAUDE.md` / `API.md` 等项目文档
5. 若用户要求，执行 `git commit` + `git push` 上传到 GitHub

## 执行流程

### 阶段 1：探索项目

1. **读取项目根目录文件**
   - 优先读取 `README.md`、`CLAUDE.md`、`API.md`、`pom.xml` / `build.gradle` / `package.json` 等入口文件
   - 若无上述文件，使用 `find` / `Glob` / `codegraph_files` 查看目录结构

2. **索引代码结构**
   - 如果项目有 `.codegraph/` 或可用 `codegraph` MCP，优先使用 `codegraph_explore`、`codegraph_files` 等工具
   - 如果 `codegraph` 未初始化且环境允许，可运行 `codegraph init -i` 建立索引后再探索
   - 否则使用 `Read` / `Agent`（Explore 类型）阅读核心类

3. **理解核心逻辑**
   - 找到核心服务类、Controller、数据模型、配置
   - 梳理调用链和数据流
   - 识别测试覆盖情况

### 阶段 2：识别待完善点

基于探索结果，从以下维度分析并记录到 `TODO.md`：

- **未实现的 TODO / 简化实现**（如 "字段级精确 Patch 尚未实现"）
- **测试覆盖缺口**（缺少单元测试的类或方法）
- **架构/性能风险**（如并发瓶颈、缺少缓存/限流/熔断）
- **文档不一致**（README 与实际代码不符、缺少 API 说明）
- **代码质量改进**（重复代码、异常处理不完善、硬编码配置）

`TODO.md` 格式要求：

```markdown
# 项目待完善点

## 1. [标题]（优先级）

**位置**: `文件路径:行号范围`

**现状**: ...

**期望**: ...

**实现建议**: ...

---
```

### 阶段 3：更新文档

根据项目实际情况，更新以下文档（如果存在且需要更新）：

- **README.md**：项目简介、快速开始、项目结构、核心流程、测试说明、配置说明
- **CLAUDE.md**：AI 助手工作指南，包括目录结构、核心流程、关键类说明、常见修改场景、测试类说明
- **API.md**：接口列表、请求/响应字段说明、错误码

更新原则：
- 不编造不存在的内容
- 保持与代码一致
- 只修改确实过时的部分，避免无意义重写

### 阶段 4：推送 GitHub（可选）

仅在用户明确要求"上传 github"、"提交"、"push" 或类似指令时执行：

1. 确认项目目录下有 `.git` 仓库；若没有，说明当前环境不支持推送
2. `git status` 查看变更（如环境允许使用 Bash）
3. `git add` 需要提交的文件（排除 `.codegraph/` 等自动生成目录）
4. `git commit` 写清晰的中文或英文 commit message
5. `git push origin <当前分支>`

**注意**: 如果在受限环境（如 subagent 或沙箱）中无法使用 Bash，或项目副本不含 `.git`，应暂停 Git 操作并向用户说明情况，而不是反复请求权限。

## 工具使用偏好

- **探索代码**：优先 `codegraph_explore`，其次 `Read`，必要时用 `Agent`（Explore 类型）做广度搜索
- **修改文件**：使用 `Read` + `Edit` / `Write`，避免直接用 `sed` / `cat` / `echo`
- **运行验证**：代码项目用 `mvn test` / `npm test` / `pytest` 等对应测试命令
- **Git 操作**：使用 `Bash` 执行 git 命令；如果环境禁止 Bash 或项目不含 `.git`，应跳过 Git 操作并向用户说明

**环境受限时的处理原则**: 如果在 subagent、沙箱或无 Bash 权限的环境中执行，优先使用 `Read`/`Edit`/`Write` 完成文件操作，将 Git 推送等需要系统命令的步骤留给用户或明确说明无法执行。

## 输出要求

完成流程后，向用户汇报：

1. **项目概况**：技术栈、核心模块、主要入口
2. **识别的待完善点**：列出 `TODO.md` 中的关键条目
3. **文档更新摘要**：哪些文档更新了，更新了什么
4. **Git 状态**：如果执行了推送，给出 commit hash 和分支

## 示例

**用户输入**：
> 熟悉下本项目吧

**执行动作**：
1. 读取 `README.md`、`pom.xml`、`CLAUDE.md`
2. 用 `codegraph_files` 和 `codegraph_explore` 探索核心类
3. 生成 `TODO.md`，记录发现的待完善点
4. 若 README / CLAUDE / API 需要更新，则更新
5. 汇报项目概况和后续建议

**用户输入**：
> 更新下 readme 和 CLAUDE 文档，然后上传 github

**执行动作**：
1. 确认当前代码变更
2. 更新 `README.md` 和 `CLAUDE.md`
3. `git add` → `git commit` → `git push origin <分支>`
4. 汇报推送结果
