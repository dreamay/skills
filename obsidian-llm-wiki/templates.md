# LLM Wiki — 初始化模板

本文件包含首次初始化 LLM Wiki Vault 时需要创建的所有文件内容。执行初始化时按此文件逐项创建。

---

## 1. 目录树

在用户指定的 Vault 根目录下创建以下目录结构：

```
{vault-root}/
├── inbox/                      # 收集箱（原始资料、草稿、附件）
├── wiki/
│   ├── concepts/               # 概念：知识卡片 + 深度文章
│   ├── comparisons/            # 对比：A vs B
│   ├── synthesis/              # 综述：跨概念横向分析
│   ├── notes/                  # 笔记：读书/课程/论文
│   └── entities/               # 实体：工具/框架/人物档案
├── SCHEMA.md                   # 指令层（AI 行为规范）
├── INDEX.md                    # 知识库入口索引 + 统计（合并了原 OVERVIEW.md）
├── CHANGELOG.md                # 变更日志
└── .user-preferences.md        # 用户偏好
```

## 2. 配置文件：.user-preferences.md

创建到 Vault 根目录下。

```markdown
---
type: system
description: 用户偏好与写作规则 — 所有 AI Agent 操作本知识库时必须遵守的核心原则
updated: {{date}}
---

# 用户偏好与写作规则

> **本文档由用户维护，所有 AI Agent 操作本知识库时必须首先阅读并遵守。**
> 本文件优先级高于 `SCHEMA.md` 中的默认行为。

---

## 核心原则

### 1. 只写一篇，不要衍生

用户问什么就写什么，**一次只生成一篇文档**。不要因为文档里提到了相关概念就自动创建一堆新文件。

- 错误做法：用户说"写一张 Docker 卡片" → 创建 Docker + Kubernetes + 容器编排 + OCI + Compose 五篇
- 正确做法：用户说"写一张 Docker 卡片" → 只创建 Docker 一篇

### 2. 链接用外部引用

文档中提到的相关概念，**用外部链接引用**（URL），不要创建新的内部笔记。

- 错误做法：`相关概念：[[Kubernetes]]、[[容器编排]]`（点进去是空文件）
- 正确做法：`延伸阅读：[Kubernetes 官方文档](https://kubernetes.io/docs/)、[Docker vs K8s](https://example.com/article)`

**例外**：只有当目标文件在知识库中**已存在**时，才使用 `[[wiki link]]`。

### 3. 总结优先，不写论文

用户要的是**一目了然的总结**，不是教科书式的长篇大论。

- 语言精简，去掉废话和套话
- 多用表格、对比、流程图（Mermaid）、代码块
- 每个章节控制在 10 句话左右，可以适当扩长
- 一篇概念卡片 800-1000 字左右，可以适当扩长

### 4. 图文并茂

能用图说清楚的不要用文字。支持的图表方式：

- **Mermaid 流程图/时序图**：展示架构和流程
- **ASCII 图**：简单结构关系
- **表格**：对比和参数罗列
- **代码块**：实际的配置或命令示例

### 5. 面向快速理解

假设用户是技术人员，有一定基础，不需要从零讲起。直接进入重点，用"是什么 → 为什么 → 怎么用"的结构。

---

## 写作风格

- 中文为主，技术术语保留英文原文（如 Namespace、Cgroups、Transformer）
- 语气：技术文档风格，客观准确
- 不加"值得注意的是""需要指出的是"等废话
- 不写"本文将介绍..."之类的开头
- 不写"总之""综上所述"之类的结尾
- 代码示例只放最核心的片段，不要贴完整配置

---

## 参考规范

完整的 AI 行为规范、Frontmatter 定义、模板结构和操作流程，请参阅 `SCHEMA.md`。

---

## 用户可自定义区域

> 以下区域用户可自行修改，AI 按最新内容执行。

### 偏好深度

- 概念卡片：精简总结（800-1000 字）
- 综述文章：中等深度（1500-2500 字）
- 来源笔记：精简总结（800-1200 字）

### 偏好图表

- 架构图：优先用 Mermaid
- 对比：优先用表格
- 流程：优先用 Mermaid 时序图或流程图

### 语言

- 正文：中文
- 代码注释：中文
- 技术术语：保留英文
```

## Frontmatter 规范

每篇笔记必须包含以下 YAML 头部：

```yaml
---
type: 笔记类型
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: []
ai_generated: true
---
```

### type 枚举

| type | 用途 | 目标文件夹 |
|------|------|-----------|
| concept | 概念卡片 | wiki/concepts/ |
| comparison | 技术对比 | wiki/comparisons/ |
| synthesis | 综合综述 | wiki/synthesis/ |
| source-note | 来源笔记 | wiki/notes/ |
| entity | 实体（人物/组织/项目） | wiki/entities/ |
| raw | 原始资料 | inbox/ |

## 概念卡片模板

**字数**：800-1000 字，精简总结

**推荐结构**：

```markdown
## 一句话定义

（1-2 句话说清楚这是什么）

## 核心原理

（Mermaid 图或表格 + 简短说明）

## 关键命令/代码

（只放最常用的 5-10 行，不贴大段配置）

## 优缺点速查

| 优点 | 缺点 |
|------|------|
| ... | ... |

## 适用场景

（3-5 个典型场景，每个一句话）

## 延伸阅读

（外部 URL，不创建内部文件）
- [标题](URL)
```

**禁止**：
- 不要写"本文将介绍..."之类的开头
- 不要在"延伸阅读"中放指向空文件的 `[[wiki link]]`
- 不要超过 1200 字

## 链接策略

### 默认：外部 URL

文档中提到的相关概念，**默认使用外部 URL 引用**：

```markdown
## 延伸阅读

- [Kubernetes 官方文档](https://kubernetes.io/docs/)
- [Docker Compose 使用指南](https://docs.docker.com/compose/)
```

### 例外：wiki link

仅在以下条件满足时使用 `[[wiki link]]`：

1. 目标文件**已存在**于知识库中
2. 用户明确要求创建关联文档

### INDEX.md 更新

新增笔记后，更新 `INDEX.md` 中对应领域的链接（这是唯一需要同步更新的内部文件）。

## 写入后清单

每次写入新笔记后，必须执行以下检查：

- [ ] 确认 frontmatter 完整且 type 值正确
- [ ] 延伸阅读使用外部 URL（不创建内部空文件）
- [ ] 更新 `INDEX.md` 中对应领域的条目
- [ ] 更新 `CHANGELOG.md`，追加变更记录
- [ ] 添加合适的 tags（从已有标签中选择）

---

## 文件已存在处理

当用户要求创建某篇笔记，但该文件**已存在**时，**禁止直接覆盖**，必须先询问用户：

> "XXX 已经存在了，你想怎么处理？"
> - **更新补充**：在现有内容基础上补充新信息
> - **完全重写**：覆盖旧内容，从头写
> - **取消**：不操作

### 判断逻辑

1. 先按命名规范计算出目标文件路径
2. 检查该路径是否已存在文件
3. 如果存在 → 询问用户选择
4. 如果不存在 → 正常创建

**禁止**：不检查就直接覆盖已有文件。

---

## 删除流程

当用户要求删除某篇笔记时，**必须按以下顺序执行全部步骤**：

1. **删除目标文件**
2. **清理 INDEX.md**：移除 `INDEX.md` 中指向已删文件的链接
3. **清理其他引用**：搜索整个 Wiki 中引用了已删文件的 `[[wiki link]]`，将其改为外部 URL 或直接移除
4. **更新 INDEX.md**：修正统计数字
5. **记录 CHANGELOG.md**：追加一条删除记录

**禁止**：只删文件不做善后。这会导致断链和统计虚高。

---

## 同步流程

当用户手动增删了文件，或说"帮我同步一下知识库"时，执行以下流程：

### 触发方式

- "帮我同步一下知识库"
- "更新一下统计信息"
- "检查一下索引"

### 同步步骤

1. **扫描实际文件**：遍历 Vault 所有目录，统计各类型文件的实际数量
2. **修正统计数字**：将 `INDEX.md` 中的计数更新为实际值
3. **补充新文件到 INDEX.md**：发现用户手动新增的文件时，**必须将其添加到 `INDEX.md` 对应的领域中**，不可删除用户创建的任何文件
4. **清理断链**：检查 `INDEX.md` 中每个 `[[wiki link]]` 是否指向实际存在的文件，移除不存在的
5. **检查笔记内断链**：搜索所有笔记中的 `[[wiki link]]`，对指向不存在文件的链接给出提醒
6. **记录 CHANGELOG.md**：追加本次同步结果

### 重要原则

- **只增不删**：同步时发现用户手动新增的文件，必须收录到 INDEX.md，绝不删除用户的文件
- **只清理断链**：只有当文件确实不存在时，才移除对应链接

---

## 日常校验规则

### 每次写入后

执行"写入后清单"中的 5 项检查。

### 每天第一次操作时

- [ ] **校验统计**：扫描实际文件，确认 `INDEX.md` 中的计数与实际一致
- [ ] **校验 INDEX.md**：检查所有链接，确认无断链、无遗漏
- [ ] 如发现不一致，自动修正并在 `CHANGELOG.md` 中记录

### 其他时间

仅在用户明确要求"同步一下知识库"时才执行完整校验，日常操作只做基础清单。

## 标签注册表

### 领域标签（domain tags）

用于标记笔记所属的知识领域，用户可根据需要扩展：

- `技术`、`架构`、`AI`、`数据库`、`DevOps`、`前端`、`后端`、`安全`、`网络`

### 状态标签（status tags）

用于标记笔记的完成状态：

- `draft` — 草稿
- `review` — 待审阅
- `done` — 完成

### 格式标签（format tags）

用于标记笔记的内容形式：

- `对比`、`综述`、`论文`、`书籍`、`课程`

### 标签规则

1. 新增标签前必须检查已有标签，避免同义重复
2. 新增标签需用户确认
3. 标签全部使用小写
4. 每篇笔记至少 1 个领域标签
