## 配置文件 1：.user-preferences.md

```markdown
---
type: system
description: 用户偏好与写作规则 — 所有 AI Agent 必须遵守的核心原则
updated: {{date}}
---

# 用户偏好与写作规则

> **本文档由用户维护，所有 AI Agent 操作本知识库时必须首先阅读并遵守。**
> 本文件优先级高于 `.ai-instructions.md` 和 `_Prompts/` 下的所有指令。

---

## 核心原则

### 1. 只写一篇，不要衍生

用户问什么就写什么，**一次只生成一篇文档**。不要因为文档里提到了相关概念就自动创建一堆新文件。

- 错误做法：用户说"写一张 Docker 卡片" → 创建 Docker + Kubernetes + 容器编排 + OCI + Compose 五篇
- 正确做法：用户说"写一张 Docker 卡片" → 只创建 Docker 一篇

### 2. 链接用外部引用，不要内部新建

文档中提到的相关概念，**用外部链接引用**（URL），不要创建新的内部笔记。

- 错误做法：`相关概念：[[Kubernetes]]、[[容器编排]]`（点进去是空文件）
- 正确做法：`延伸阅读：[Kubernetes 官方文档](https://kubernetes.io/docs/)、[Docker vs K8s](https://example.com/article)`

**例外**：只有当用户明确要求、或某个概念与当前话题高度相关且知识库中已有内容时，才使用 `[[wiki link]]`。

### 3. 总结优先，不写论文

用户要的是**一目了然的总结**，不是教科书式的长篇大论。

- 语言精简，去掉废话和套话
- 多用表格、对比、流程图（Mermaid）、代码块
- 每个章节控制在 10 句话左右，可以适当扩长
- 一篇知识卡片 800-1000 字左右，可以适当扩长

### 4. 图文并茂

能用图说清楚的不要用文字。支持的图表方式：

- **Mermaid 流程图/时序图**：展示架构和流程
- **ASCII 图**：简单结构关系
- **表格**：对比和参数罗列
- **代码块**：实际的配置或命令示例

### 5. 面向快速理解

假设用户是技术人员，有一定基础，不需要从零讲起。直接进入重点，用"是什么 → 为什么 → 怎么用"的结构。

## 文档结构模板（精简版）

知识卡片推荐结构（不是每个章节都必须）：

```
## 一句话定义
（一段话说清楚这是什么）

## 核心原理
（Mermaid 图 或 ASCII 图 + 简短说明）

## 关键命令/代码
（只放最常用的，不要贴大段配置）

## 优缺点速查
（表格形式，一目了然）

## 适用场景
（3-5 个典型场景，一句话描述）

## 延伸阅读
（外部链接，不创建内部文件）
```

---

## 写作风格

- 语气：技术文档风格，客观准确，不加"值得注意的是""需要指出的是"等废话
- 中文为主，技术术语保留英文原文（如 Namespace、Cgroups）
- 不写"本文将介绍..."之类的开头
- 不写"总之""综上所述"之类的结尾
- 代码示例只放最核心的片段，不要贴完整配置

---

## 禁止行为清单

- [x] 用户没要求时，不要自动创建关联文档
- [x] 不要在"相关概念"中放指向空文件的 `[[wiki link]]`
- [x] 不要写超过 1200 字的知识卡片
- [x] 不要写"本文将..."、"接下来我们..."等教科书式开头
- [x] 不要在没有代码必要的地方强行加代码块
- [x] 不要重复说同一件事

---

## 用户可自定义区域

> 以下区域用户可自行修改，AI 按最新内容执行。

### 偏好深度

- 知识卡片：精简总结（800-1000 字）
- 技术文章：中等深度（1500-2500 字）
- 读书笔记：精简总结（800-1200 字）

### 偏好图表

- 架构图：优先用 Mermaid
- 对比：优先用表格
- 流程：优先用 Mermaid 时序图或流程图

### 语言

- 正文：中文
- 代码注释：中文
- 技术术语：保留英文
```

## 内容生成要求

### 知识卡片（最高频）

**字数**：600-1000 字，精简总结，不是论文

**推荐结构**：
1. 一句话定义（1-2 句）
2. 核心原理（Mermaid 图 或 表格 + 简短说明）
3. 关键命令/代码（只放最常用的片段）
4. 优缺点速查（表格形式）
5. 适用场景（3-5 个，一句话描述）
6. 延伸阅读（外部 URL，不创建内部文件）

**禁止**：
- 不要写"本文将介绍..."之类的开头
- 不要在"相关概念"中放指向空文件的 `[[wiki link]]`
- 不要超过 1200 字

### 读书笔记

- 800-1200 字，提取核心观点，不是章节复述
- 加入分析视角和跨领域关联
- 生成可执行的行动清单

### 技术对比

- 表格横向对比，给出场景化推荐
- 引用最新数据和社区活跃度

### 周报/回顾

- 基于本周会议记录和日记自动汇总
- 提取关键进展、风险点、下周重点
- 量化成果

### 博客草稿

- 开头抓人，核心论点明确
- 包含实际案例或代码
- 结尾有总结和行动建议

## 链接策略

### 内部链接 `[[wiki link]]` 的使用条件

只在以下情况使用内部 `[[wiki link]]`：

1. 目标文件**已经存在**于知识库中
2. 用户明确要求创建关联文档
3. 某概念与当前话题高度相关，且知识库中已有该内容

### 外部链接的使用

文档中提到的相关概念，**默认用外部 URL 引用**：

```markdown
## 延伸阅读

- [Kubernetes 官方文档](https://kubernetes.io/docs/)
- [Docker Compose 使用指南](https://docs.docker.com/compose/)
```

### MOC 更新

新增笔记后，更新对应的 `07-MOC/` 索引文件（这是唯一需要同步更新的内部文件）。

## 文件删除流程

当用户要求删除某篇笔记时，**必须按以下顺序执行全部步骤**，不可只删文件了事：

1. **删除目标文件**
2. **扫描并清理 MOC 索引**：遍历 `07-MOC/` 下所有索引文件，移除指向已删文件的 `[[wiki link]]`
3. **扫描并清理其他笔记**：搜索整个 Vault 中引用了已删文件的 `[[wiki link]]`，将其改为外部 URL 或直接移除
4. **更新统计数字**：修正 `CONTEXT.md` 和 `07-MOC/Vault总览.md` 中的文件计数
5. **记录删除日志**：在 `07-MOC/最近更新.md` 的摘要区追加一条删除记录

**禁止**：只删文件不做善后。这会导致 MOC 断链和统计虚高。

---

## 文件已存在处理

当用户要求创建某篇笔记，但该文件**已存在**时，**禁止直接覆盖**，必须先询问用户：

> "XXX 知识卡片已经存在了，你想怎么处理？"
> - **更新补充**：在现有内容基础上补充新信息
> - **完全重写**：覆盖旧内容，从头写
> - **取消**：不操作

### 判断逻辑

1. 先按命名规范计算出目标文件路径
2. 检查该路径是否已存在文件
3. 如果存在 → 询问用户选择
4. 如果不存在 → 正常创建

**禁止**：不检查就直接覆盖已有文件。

---

## 知识库同步

当用户手动增删了文件，或说"帮我同步一下知识库"时，执行以下流程：

### 触发方式

- "帮我同步一下知识库"
- "更新一下统计信息"
- "检查一下 MOC 索引"

### 同步步骤

1. **扫描实际文件**：遍历 Vault 所有目录，统计各类型文件的实际数量
2. **修正统计数字**：将 `CONTEXT.md` 和 `07-MOC/Vault总览.md` 中的计数更新为实际值
3. **补充新文件到 MOC**：发现用户手动新增的文件时，**必须将其添加到对应的 MOC 索引中**，不可删除用户创建的任何文件
4. **清理 MOC 断链**：遍历 `07-MOC/` 下所有索引文件，检查每个 `[[wiki link]]` 是否指向实际存在的文件
5. **检查笔记内断链**：搜索所有笔记中的 `[[wiki link]]`，对指向不存在文件的链接给出提醒
6. **更新最近更新**：在 `07-MOC/最近更新.md` 摘要区记录本次同步结果

### 重要原则

- **只增不删**：同步时发现用户手动新增的文件，必须收录到 MOC，绝不删除用户的文件
- **只清理断链**：只有当文件确实不存在时，才从 MOC 中移除对应链接
```

## AI 操作规范

### 核心原则（详见 `.user-preferences.md`）

- **只写一篇**：用户问什么就写什么，不衍生创建关联文件
- **外链引用**：相关概念用外部 URL 引用，不创建内部空文件
- **精简总结**：知识卡片 600-1000 字，图文并茂，不写论文

### 写入新笔记后必须执行

1. 确认 frontmatter 完整且 type 值正确
2. 延伸阅读使用外部 URL（不创建内部空文件）
3. 更新 07-MOC/ 中对应的索引文件
4. 更新 `07-MOC/最近更新.md` 摘要区
5. 添加合适的 tags（从已有标签中选择）

### 读取操作

- 搜索笔记时优先按 type 和 tags 过滤
- 读取 07-MOC/ 可以快速了解某个领域有哪些笔记
- 读取 `.user-preferences.md` 获取用户写作偏好（最高优先级）
- 读取 `.ai-instructions.md` 获取行为规范
- 读取 `_Prompts/` 下的指令文件了解如何生成特定类型内容

### 文件存放规则

- 明确知道属于哪个领域 → 直接放入对应文件夹
- 不确定分类 → 先放入 00-Inbox，后续整理
- 技术知识卡片 → 优先放入 03-学习/技术专题/ 对应子目录
- 通用概念卡片 → 放入 03-学习/知识卡片/

当用户要求生成读书笔记时，按以下流程操作：

- 用户提供阅读笔记的草稿，AI 整理成标准格式

- 行动清单要具体可执行，不是空话

## 周回顾生成

## 月度复盘生成

   - 结尾：总结核心观点 + 给出行动建议

## 写作原则

1. 初稿完成 → 状态改为 `draft`
2. 用户审阅修改 → 状态改为 `review`
